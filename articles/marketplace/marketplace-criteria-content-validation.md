# <a name="azure-marketplace-listing-guidelines"></a>Azure Marketplace の掲載に関するガイドライン

この記事では、Azure Marketplace に新しいオファーとサービスを掲載する際の要件とガイドラインについて説明します。 すべてのオファーは、以下に示されている[すべての掲載要件](#listing-requirements-for-all-offers)を満たす必要があります。 右側のリンクを使用して、特定の掲載の種類に関する要件とチェックリストに移動できます。

## <a name="listing-requirements-for-all-offers"></a>すべてのオファーの掲載要件

| いいえ。 | 掲載する要素 | 基本要件 | 最適な要件 |
|:--- |:--- |:--- |:--- |
| 1 | オファーのタイトル | ソリューション オファリングについて簡単に説明する。 パートナーの Web サイトでのソリューションのオンライン プロモーションと一致している。 | 主要な検索語句が含まれている。 |
| 2 | ロゴ | ロゴが正しく表示される。  | ロゴが正しく表示される。  |
| 3 |  オファーの説明 | <ul><li> 2 から 3 段落が含まれている。</li><li>ソリューション オファリングがひとめで簡単に理解できる。</li><li>スペルと文法の誤りがない。</li><li>包括的で、対象ユーザー、ユーザーの種類、それが役立つ理由 (価値提案) をとらえている。</li><li>段落形式の、理解しやすい短文での語り口になっている。</li></ul> | <ul><li> 対象業界がおおよそ示されている (該当する場合)。</li><li>各段落の見出しが、後続の内容をまとめた 1 つの文または語句になっている適切なスタイルで書式設定されていて、適切な場合には、箇条書きを使用して主なメリットを強調する。 目的は、見やすい形式で読者がひとめでオファーを理解し、長い段落を読まずに済むようにすることです。</li><li>各段落の間にスペースがある。 車のパンフレットのように読める。 つまり、包括的で、技術的な専門用語を使用せずに、オファリングを簡潔に説明します。 | 
| 4 | カテゴリと業界 | <ul><li>カテゴリとサブカテゴリはオファーの機能と一致する必要があります </li><li>オファーの機能に対応しないカテゴリまたはサブカテゴリは選択しないでください。 </li></ul> | <ul><li>プライマリ カテゴリとセカンダリ カテゴリ を含む、最大 2 つのカテゴリを選択します (省略可能)。</li><li>プライマリ カテゴリ、セカンダリ カテゴリ、またはその両方についてそれぞれ最大 2 つのサブカテゴリを選択します。 サブカテゴリを選択しない場合でも、オファーは選択したカテゴリで検出できます。</li></ul> |
| 5 | イメージ | <ul><li>画像の要件がパートナー センターに記載されていること。</li><li>スクリーンショットに含まれるテキストが読みやすく、画像がはっきり見えること。 | ソリューション オファリングがひとめで簡単に理解できる。 |
| 6 | ビデオ | <ul><li>ビデオは必須ではないが、提供する場合はエラーなく再生される必要がある。</li><li>提供する場合は、移行ソリューションのデモである場合を "*除き*"、競合企業に言及しない。 |<ul><li>長さは 3 分以上であることが望ましい。</li><li>ソリューション オファリングがビデオ コンテンツを通して容易に理解できる。</li><li>ソリューションの機能のデモを提供する。 |
| 7 | 掲載状態 (掲載オプション) | <ul><li>以下のいずれかの種類のラベルが付いている必要がある。 <ul><li>*連絡する*</li><li>"*試用版*"/"*試用版を入手する*"/"*試用を開始する*"/"*体験版*"</li><li>"*今すぐ購入*"/"*今すぐ入手*"</li></ul></ul> | 顧客が、次のステップが何かを容易に理解できる。 <ol><li>試用版を試す。</li><li>今すぐ購入する。</li><li>概念実証 (POC)、評価、または説明会を手配するために電子メールまたは電話番号で問い合わせる。</li></ol> | 
| 8 | ソリューションの価格 | ソリューションの価格のタブまたは詳細があり、価格がパートナー ソリューション オファリングの現地通貨で示されていること。 | 顧客に複数の選択肢を与えるため、レベル別の価格で複数の課金オプションを使用できるようにすること。 |
| 9 | 詳細情報 | 下部 (左側の Azure Marketplace のリンクではなく、説明の下) にあるリンクが、ソリューションに関する詳細情報に移動できるようになっていて、一般にアクセス可能で、正常に表示される。 |    パートナーのホーム ページだけでなく、特定の項目 (パートナー サイトの特定ページなど) へのリンクがある。 | 
| 10 | ソリューションのサポートとヘルプ | 以下のうち、少なくとも 1 つへのリンク: <ul><li>電話番号</li><li>メール サポート</li><li>チャット エージェント</li><li>コミュニティ フォーラム |<ul><li>すべてのサポート方法が一覧表示されている。</li><li>"*試用版*" や "*体験版*" 期間中は有料サポートが無料で提供される。 |
| 11 | 法的情報 | ポリシーや条件がパブリック URL で参照できる。 |  |
|||

## <a name="saas-application-requirements"></a>SaaS アプリケーションの要件

| いいえ。 | 掲載する要素 | 基本要件 | 最適な要件 | 
|:--- |:--- |:--- |:--- |
| 1 | オファーのタイトル |<ul><li>小文字の英数字、ダッシュ、アンダースコアのみで構成しなければならない。 公開後にタイトルを変更することはできない。</li><li>ソリューション オファリングについて説明する。</li><li>パートナーの Web サイトでのソリューションのオンライン プロモーションと一致している。 | 主要な検索語句が含まれている。 |
| 2 | 技術情報:構成 |<ul><li>サービスとしてのソフトウェア (SaaS) アプリの場合、アプリを掲載するだけか、顧客が Azure を通してアプリを購入できるようにするかどうかを選択する。</li><li>オファー入手のボタンに表示するテキストを選択する:"*無料*"、"*無料試用版*"、"*問い合わせ*"。</li><li>アプリが次の技術を利用している場合は、ポップアップ ウィンドウで該当する製品を 1 つだけ選択する。Cortana Intelligence、Power BI ソリューション テンプレート、Power Apps。 |  |
| 3 | 体験版 | *[はい]* または *[いいえ]* を選択します | 顧客が、次のステップが何かを容易に理解できる。 <ol><li>試用版を試す。</li><li>今すぐ購入する。</li><li>POC、評価、または説明会を手配するために電子メールまたは電話番号で問い合わせる。</li></ol> |
| 4 | オンライン ストアの詳細:オファーの概要 | アプリの検索ページに表示される (最大 100 文字)。 | |
| 5 | オンライン ストアの詳細:業界 | 業界 (最大 2):アプリに最もよく当てはまり、アプリを適用できる業界を選択します。 | |
| 6 | オファーの説明 | <ul><li>\<p>、\<em>、\<ul>、\<li>、\<ol>、ヘッダー タグなど、単純な HTML を使用できます。 最大 3,000 文字。</li><li>2 から 3 段落が含まれている。</li><li>ソリューション オファリングがひとめで簡単に理解できる。</li><li>説明が包括的で、対象ユーザー、ユーザーの種類、それが役立つ理由 (価値提案) をとらえている。</li><li>段落形式の、理解しやすい短文での語り口になっている。 | <ul><li>対象業界がおおよそ示されている (適切な場合)。</li><li>各段落の見出しが、後続の内容をまとめた 1 つの文または語句になっている適切なスタイルで書式設定されていて、適切な場合には、箇条書きを使用して主なメリットを強調する。 目的は、見やすい形式で読者がひとめでオファーのことを理解し、複数の段落を読まずに済むようにすることです。</li><li>各段落の間にスペースがある。 車のパンフレットのように読める。 つまり、包括的で、技術的な専門用語を使用せずに、オファリングを簡潔に説明します。 | 
| 7 | マーケティングの成果物 | ロゴが正しく表示される。 | <ul><li>ロゴ:小 (48 x 48 px、省略可能)、中 (90 x 90 px、省略可能)、大 (216 x 216 から 350 x 350 px、必須)。</li><li>スクリーンショット (最大 5):解像度 1280 x 720 ピクセルの .PNG 画像が必要です。 |
| 8 | カテゴリと業界 | <ul><li>カテゴリが、ソリューション オファリングの機能と一致している。</li><li>ポップアップ ウィンドウで、少なくとも 1 つの項目を選択する。</li><li>ソリューションが適合しないカテゴリは選択しない。</li><li>最適な業界、または (最適な業界でない場合は) すべての業界が選択されている。 | 最大 3 つのカテゴリを選択 (該当する場合)。 |
| 9 | リードの管理 | 潜在顧客を保存するシステムを選択する。 お使いの CRM システムに接続するには、[潜在顧客の獲得](./partner-center-portal/commercial-marketplace-get-customer-leads.md)に関するページを参照してください。 |  |
| 10 | 連絡先:ソリューションのサポートとヘルプ | <ul><li>エンジニアリング連絡先の名前:アプリのエンジニアリング関連の連絡先の名前。 この連絡先が、Microsoft から技術的な通信を受け取ることになります。</li><li>エンジニアリング連絡先の電子メール:アプリのエンジニアリング関連の連絡先のメール アドレス。</li><li>エンジニアリング連絡先の電話:エンジニアリング関連の連絡先の電話番号。 [ISO の電話番号表記](https://en.wikipedia.org/wiki/E.123)がサポートされています。</li><li>サポート連絡先の名前:アプリのサポート連絡先の名前。 この連絡先が、Microsoft からサポート関連の通信を受け取ることになります。</li><li>サポート連絡先の電子メール:アプリのサポート連絡先のメール アドレス。</li><li>サポート連絡先の電話:サポート連絡先の電話番号。 [ISO の電話番号表記](https://en.wikipedia.org/wiki/E.123)がサポートされています。</li><li>サポート URL:サポート ページの URL。 | <ul><li>すべてのサポート方法が一覧表示されている。</li><li>"*試用版*" や "*体験版*" 期間中は有料サポートが無料で提供される。 |
| 11 | 法的情報 |<ul><li>プライバシー ポリシーの URL:CPP のプライバシー ポリシー URL のフィールドにあるアプリのプライバシー ポリシーへの URL。</li><li>利用規約:アプリの使用条件。 顧客は、アプリを試す前にこれらの条件を承諾する必要があります。 |    ポリシーや条件がパブリック URL サイトで参照できる。 |
|||

## <a name="consulting-offer-requirements"></a>コンサルティング オファーの要件

| いいえ。 | 掲載する要素 | 基本要件 | 最適な要件 |
|:--- |:--- |:--- |:--- |
| 1 | オファーのタイトル  |  サービスの種類と期間を、"*名前:期間の種類*" (例: "オファー エンゲージメント:1 週間の概念実証") の形式で明確に記載すること。  | <ul><li>発行元の名前を繰り返さない。</li><li>タイトルには説明で既に利用できるタイトル スタイルを反映させる。 長いタイトルは使用しない。 | 
| 2 | オファーの説明  |<ul><li>[Microsoft の製品名](https://www.microsoft.com/en-us/legal/intellectualproperty/trademarks/en-us.aspx)が適切に使用されていることを確保する。</li><li>"*価格:概算*" としてマークされたオファーには、上部または下部のいずれかに、変動要因 (クライアントへの移動費、移行されるサーバー数など) を説明する注記を付ける必要がある。</li><li>オファーの種類ごとに、次のような説明の要件がある。 <ul><li>ブリーフィングには、説明するトピックについての情報が記載された、少なくとも 4 つか 5 つの箇条書きが必要。</li><li>すべてのワーク ショップに議題が必要。</li><li>ワークショップの期間に応じて、すべての議題を日単位または週単位で分けする必要がある。</li><li>評価、POC、トレーニング以外のワーク ショップ、実装のオファーには成果物が必要。</li><li>トレーニング ワークショップには、成果物は不要だが、取り上げられるトピックを示したより詳細な議題が必要。</li></ul></ul> | <ul><li>どのオファーにも議題と成果物があること。</li><li>オファーの最上部セクションに、サービスを提供する企業に関する状況を示した段落が含まれていること。</li><li>オファーの最上部セクションとして、サービス自体の価値に関する段落が含まれていること。 |
| 3 | Markdown 書式設定  | すべてのオファーで、HTML への変換時にオファーが正しくレンダリングされるように Markdown 書式設定を使用すること。  |  |
| 4 | カテゴリと業界 | カテゴリは関係がない。 |  | 
| 5 | 掲載状態 (掲載オプション) | 自動的に "*問い合わせ*" と表示される。 |  | 
| 6 | ソリューションのサポートとヘルプ  | サポートとヘルプは不要。  | | 
| 7 | プライバシー ポリシーと利用規約の提供  |  ポリシーや条件は不要。  | | 
| 8 | サービスの種類 | サービスの種類とタイトルを確実に一致させる。 | |
| 9 | コンピテンシー | 次の少なくとも 1 つでコンピテンシーがあること。<ul><li>アプリケーション開発</li><li>アプリケーションの統合</li><li>アプリケーション ライフサイクル管理</li><li>クラウド プラットフォーム<li>データ分析</li><li>データ センター</li><li>データ プラットフォーム</li><li>DevOps |
| 10 | 製品 | Azure 製品であること。 | |
| 11 | 国/リージョン | 国/リージョンと選択した通貨を確実に一致させる。 | |
| 12 | 詳細情報  | <ul><li>下部 (左側の Azure Marketplace のリンクではなく、説明の下) にあるリンクが、ソリューションに関する詳細情報に移動できるようになっていて、一般に公開されており、正常に表示されている。</li><li>リンクには "フレンドリ名" が付いていて、どのダウンロードでもファイル名としては表示されないこと。 | |
